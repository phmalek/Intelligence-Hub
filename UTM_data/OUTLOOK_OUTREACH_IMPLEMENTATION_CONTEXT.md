## What We Discussed Next
The next feature is a local Outlook-based outreach workflow.

### Intended use case
Example: UTM adoption follow-up.

The user wants to:
1. choose markets by UTM adoption status
2. select which markets to contact
3. choose recipients from the address book
4. choose or write an email template
5. send emails to those markets
6. track which emails were sent
7. press a `Sync` button later
8. inspect Outlook inbox / sent threads for replies
9. mark which markets replied and when

## Key Technical Decision Already Reached

### Environment constraint
The project was being worked on inside Linux / WSL.

Conclusion:
- `pywin32` **cannot** run inside WSL/Linux Python
- `pywin32` requires **native Windows Python** because it depends on Windows COM

The user said they will move the project to the Windows machine.

### New preferred implementation path
Once running on native Windows:
- use `pywin32`
- connect directly to desktop Outlook via COM

This is the preferred path for true local Outlook integration.

## Recommendation Already Given
For email tracking, **do not use CSV as the primary tracking store**.

Use:
- `SQLite` for workflow state and message tracking
- CSV only for import/export if needed

Reason:
This feature needs durable tracking for:
- send events
- recipients
- markets
- subject/body used
- timestamps
- Outlook message ids / conversation ids
- reply sync status
- last reply time
- reply snippets / sender info

CSV is too weak for that workflow.

## Recommended Architecture

### Core rule
Send **one email per market**.

Do **not** send one email covering multiple markets if reply tracking matters.

Reason:
- reply matching becomes unreliable
- market-level follow-up becomes ambiguous
- syncing state per market becomes harder

### High-level flow
1. filter markets from the `UTM Adoption` page by status
2. choose selected markets
3. resolve recipients from address book
4. render a market-specific email from a template
5. preview before send
6. send one email per market through Outlook desktop
7. store message metadata in SQLite
8. later, click `Sync Replies`
9. scan Outlook for replies using conversation/message identifiers
10. update tracking state in SQLite

## Strong Recommendations For Implementation

### 1. Use SQLite
Suggested local DB file, for example:
- `UTM_data/outreach_tracking.db`

### 2. Track messages by Outlook identifiers
Store as much of this as available:
- Outlook EntryID
- ConversationID
- Subject
- Sent timestamp
- Recipient list
- Market
- Workflow type, e.g. `UTM Adoption`
- Template name/version

### 3. Recipient logic must be explicit
A market can have multiple contacts and sections.
Define a clear rule such as:
- primary recipients: `PlanIt Champion`
- fallback: `Market Key Contact`
- optional CC: `Digital Contact`, `Search Contact`

This should be user-configurable, not hard-coded forever.

### 4. Start simple on reply sync
First implementation should:
- check whether a reply exists for each sent market email
- capture:
  - `replied` boolean
  - `last_reply_at`
  - `last_reply_from`
  - optional short snippet

Do not overbuild inbox parsing in v1.
